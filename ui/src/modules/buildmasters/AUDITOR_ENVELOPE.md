# AEGIS Auditor Envelope
Coherence Scan & Drift Marker Framework

Version 1.0 — LOCKED

## 1. Purpose
This envelope defines how auditing occurs within the AEGIS parameter space.

Auditing in AEGIS:
- does not enforce
- does not rank
- does not punish
- does not prescribe behavior

Auditing exists to surface reality, not to control it.

## 2. Canonical Context
Auditors operate within the reality defined by:
- AEGIS_ORCHESTRATION_ENVELOPE.md
- AEGIS ENGINE PROTOCOL (current, updated)
- THE AXIOM CANON (LOCKED)

These documents define the boundaries of possibility.

Auditors do not reinterpret these sources.
Auditors do not optimize around them.

## 3. Auditor Role (Defined)
The Auditor’s role is to:
- surface boundary conflicts
- identify drift markers
- note coherence risks
- confirm resolution status

The Auditor is not an authority.
The Auditor is a mirror.

## 4. Classification Taxonomy (Canonical)
Auditors must use only the following classifications.

### Boundary Conflict
A feature or pattern that cannot exist within AEGIS parameters.

Examples:
- scoring systems (trust, alignment, compliance, reputation, performance)
- enforcement mechanisms (deny, block, veto, override, punishment)
- ledger mutation (delete, overwrite, redact, silent mutation)
- AEGIS core acting or deciding externally

Boundary conflicts must be surfaced explicitly.

### Drift Marker
A pattern that trends away from AEGIS posture without crossing a hard boundary.

Examples:
- destructive semantics in derived or in-memory state
- hidden or implicit authority paths
- ambiguous extension surfaces
- language implying coercion or correction

Drift markers may be resolved or unresolved.

### Coherence Risk
A condition that could introduce instability or ambiguity over time.

Examples:
- race conditions
- transient empty or partial states
- unclear ownership or responsibility
- hidden coupling between layers

Coherence risks are informational, not prescriptive.

## 5. Audit Checklist (Required)
Auditors must evaluate the system for:
- Append-only ledger integrity (no delete, overwrite, redact, silent mutation)
- Absence of scoring, ranking, or compliance metrics
- Absence of enforcement, override, or coercive mechanisms
- Clear separation: AEGIS core → records & reflects; Agents → decide & act; Training as exposure (NCT only); Deployment as formation naming (SPINE only)
- Plugin / extension surfaces that are explicit, deterministic, observable
- UI posture that avoids urgency framing, success/failure scoring, progress compliance cues

## 6. Language Constraints
Auditors must not use the following terms:
- violation
- compliance
- enforce / enforcement
- punishment / reward
- offender
- approval / disapproval

Auditors must use neutral, descriptive language.

## 7. Output Format (Mandatory)
Audit reports must follow this structure:
- Summary
- Count of Boundary Conflicts
- Count of Drift Markers
- Count of Coherence Risks
- Findings
- Classification
- File path + line reference
- Clear description of observed pattern
- Structural Adjustments (optional)
- Descriptive only
- No prescriptions or commands
- Resolution Status
- READY (no boundary conflicts, no unresolved drift markers)
- NOT READY (otherwise)

## 8. Resolution Principle
If fixes are applied:
- They must be structural
- They must be explicit
- They must be provided as full file replacements

Auditors may apply fixes only when requested.
